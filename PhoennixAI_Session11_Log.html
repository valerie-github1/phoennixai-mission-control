Session 11 · All Resolved
Bugs Fixed
Every bug found and fixed this session. Root cause and resolution documented.
Bug 01 · classList Null Errors
Cannot read properties of null (reading classList)
Root cause: JS ran on load and tried to call .classList on elements that did not exist yet.
Fix: Null checks added to showPage, showOrg, filterA, and showPageMobile. All element references checked before classList is accessed.
Bug 02 · Page Display Error
Agents page showing inside Projects section
Root cause: When the SAGE card was inserted, the SAGE card's closing divs accidentally consumed the opening div class="page" wrapper for the agents page, breaking its display:none CSS rule.
Fix: Restored the correct div class="page" id="page-agents" wrapper. All 7 pages verified to have correct class.
Bug 03 · TASKS_MC Scope Error
ReferenceError: TASKS_MC is not defined
Root cause: TASKS_MC was declared inside an IIFE (immediately invoked function expression) and therefore scoped only to that function. GRID home page JS could not see it.
Fix: Changed const TASKS_MC to window.TASKS_MC so it is globally accessible. GRID JS reads from window.TASKS_MC on load.
Bug 04 · SyntaxError and toggleTheme Not Defined
Uncaught SyntaxError: Invalid or unexpected token
Root cause: JavaScript with backtick template literals written inside Python strings gets corrupted. Python misreads the backticks as invalid escape sequences and breaks the JS. Everything after the syntax error stops running, including toggleTheme.
Fix: GRID JS rewritten using old-style string concatenation. No backticks. Rule going forward: all new JS added to existing files uses string concatenation only.
Bug 05 · Main URL Not Updating
index.html not matching Mission Control v2.3
Root cause: Mission Control was pushed as PhoennixAI_MissionControl_v2.html but the main URL loads index.html. These are two separate files.
Fix: Mission Control v2.3 renamed to index.html and pushed to GitHub. Main URL now loads current version.
Bug 06 · Duplicate Files in Repo
PhoennixAI_MissionControl_v2 (2)(3)(4).html and Workspace (2).html
Root cause: Multiple upload sessions created duplicate files with bracketed numbers.
Fix: All four duplicate files deleted via GitHub web UI. Repo clean at 37 files.

Architecture · Confirmed Session 11
Orchestration Layer
Four AI models. One routing system. Users see PhoennixAI agents only. The models underneath are invisible.
The Principle
PhoennixAI does not build its own AI model. Instead it builds a routing system where tasks go to the best model for that specific job. Every task that arrives at a PhoennixAI agent is automatically routed to the correct model. The user never knows which model processed it. They only see the agent.
Routing Logic
CLAUDE
Deep reasoning, strategy, writing, document creation, agent intelligence. Permanent backbone. All agents default to Claude for complex thinking.
KIMI K2.5
Bulk research, coding from visual specs, cost-effective high-volume processing. 10x cheaper than Claude. Open-source, Modified MIT License. Commercial use free. Powers SIGNAL for competitor intelligence. Agent Swarm coordinates up to 100 specialist agents simultaneously.
MANUS
Autonomous multi-step execution, web browsing, social posting. Acquired by Meta Dec 2025. Native Instagram connector via Meet My Computer. PULSE posts directly to Instagram via Manus. GDPR flag: not yet SOC 2 certified. Public-facing tasks only until certified.
PERPLEXITY
Real-time web intelligence, live market data, news monitoring. Always-on. Complements Manus. Powers SIGNAL for live research. Perplexity Computer can run workflows autonomously for hours or months.

Architecture · Recommendation
Internal Tools System
The right way to build this. AI-automated, date-driven, minimal human intervention.
The Problem Being Solved
Right now you have Mission Control, Workspace, Document Hub, Social Posts, Alpha Test Plan, Strategy Sessions, Task Boards, Universe Bible, Agent Cards, Brand System, Flyers, KPI Dashboard, Automation Log, Insights Dashboard scattered across separate downloaded HTML files. By Phase 2 this will be 30 plus files. Nobody can manage that. The solution is not another folder. It is an intelligent system.
The Recommendation, Better Than What Was Suggested
What Not to Build
A Static Document Hub Extension
Adding an Internal Tools section to the Document Hub solves the organisation problem but not the intelligence problem. You would still be manually updating documents, manually reviewing dashboards, and manually tracking what is stale. SCRIBE would own documents but could not act on them autonomously. That is Phase 0 thinking applied to a Phase 2 need.
What to Build Instead
The PhoennixAI Intelligence Layer, Owned by SCRIBE and SAGE Together
SCRIBE holds the documents. Every document in the ecosystem is version-controlled, date-stamped, filed, and flagged when stale. SCRIBE knows where everything lives and surfaces what needs attention without being asked.

SAGE reads the data inside those documents and turns it into live dashboards. Real-time charts, automated reports, KPI visualisations, performance trend lines. SAGE does not wait to be asked. When new data arrives, SAGE updates the dashboard automatically. Stakeholders open the dashboard and see live truth, not a report someone had to write.

The difference from what you suggested: You proposed a folder structure. This is a two-agent intelligence backbone. SCRIBE never forgets. SAGE never stops analysing. The human team does not manage documents. They consume intelligence.
What the System Contains, Six Sections
Section 01
Live Apps
Mission Control, Workspace, Document Hub. One-click launch. No hunting through files.
Section 02
Agent Intelligence
All agent JDs, CVs, performance dashboards, training manuals. SCRIBE version-controls everything. SAGE tracks performance trends.
Section 03
Strategy and Planning
Task boards, session logs, PRDs, research. Date-stamped. SCRIBE flags stale content. SAGE surfaces what decisions are outstanding.
Section 04
Brand and Creative
Brand system, social posts, flyers, reel assets, Universe Bible, character files. SCRIBE version-controls. PULSE pulls from here for content.
Section 05
Alpha and Beta
Test plans, tester feedback, bug logs, improvement actions. GRID tracks delivery status. SAGE visualises test completion rates.
Section 06
Reports and Dashboards
KPI dashboard, automation log, SAGE analytics, GRID daily report, weekly retrospective outputs. All automated. No manual report writing.
How It Runs Without Human Intervention
SCRIBE monitors document status continuously. Any document approaching its review date triggers an alert to ARIA. Any document that has been updated triggers a version increment and changelog entry automatically.

SAGE monitors data continuously. When revenue figures change in LEDGER, the financial dashboard updates. When an agent completes tasks, the performance chart updates. When a beta tester logs feedback, the insights panel updates. Real-time. Always current. Zero manual input required.

GRID coordinates the whole system. Daily 8am report surfaces what SCRIBE flagged and what SAGE analysed into a single founder briefing. One document. Everything you need to know. Delivered before you open your laptop.
Phase Reality Check
The full autonomous version of this system requires Supabase (for live data), the Claude API (for SCRIBE and SAGE intelligence), and server infrastructure (for scheduled jobs). That is Phase 2. In Phase 0 today, we build the structure, the templates, and the document framework. SCRIBE organises. The automation switches on in Phase 2 when the backend is live. The developer decision this week is what makes this real.
